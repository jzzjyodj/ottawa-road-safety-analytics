# Ottawa Road Safety — Natural Language Query Interface

Ask plain English questions about 94,406 Ottawa traffic collisions and get back SQL, a results table, and an optional map which are all generated by a Groq-hosted LLM and validated.

---

## Project Background

Early in my data science career I worked alongside finance and marketing colleagues who needed data but depended on analysts to write queries for them. Simple questions created bottlenecks and slowed down their work. I wanted to learn how to apply LLMs in a practical context and building a system where non-technical users could ask plain English questions and get data-backed answers felt like the right problem to solve.

## System Architecture

The pipeline is a two-stage LangGraph graph wrapped in a Streamlit UI.

1. **`retrieve_schema_node`** — embeds the user question with `all-MiniLM-L6-v2` and queries an in-memory ChromaDB collection of 8 schema chunks (column groups). Returns the top 5 most relevant chunks as retrieved context.
2. **`text_to_sql_node`** — injects the retrieved context plus the full static schema into the prompt template, then calls Groq (`llama-3.3-70b-versatile`) to generate DuckDB SQL.
3. **`validate_sql_node`** — runs five structural checks (unsafe keywords, allowed table, single statement, semicolon strip, UNSUPPORTED QUERY pass-through) before any database call.
4. **DuckDB execution** — the validated SQL runs against a single denormalised `collisions_clean` table of 94,406 rows and 37 columns.
5. **Streamlit + Folium** — results render as a table, and if the result includes `lat`/`long` columns, a Folium map of the collision locations is drawn.

See [docs/architecture.md](docs/architecture.md) for the Mermaid architecture diagram and [docs/architecture_notes.md](docs/architecture_notes.md) for design decisions and known limitations.

## Eval Results

Evaluation uses `deepeval`'s GEval metric with a custom Groq judge (`meta-llama/llama-4-scout-17b-16e-instruct`) running against 15 hand-labeled test cases. Production SQL generation uses `llama-3.3-70b-versatile`.

**Prompt v1**

| Metric | Average Score | Pass Rate |
|---|---|---|
| Answer Relevancy | 0.89 | 93.33% |
| SQL Faithfulness | 0.99 | 100.00% |

The single relevancy failure is Q10 ("How many accidents were due to drunk driving?"). The pipeline correctly returns `UNSUPPORTED QUERY` because the schema has no impairment column, but the judge does not consistently honour the GEval special-case instruction ("UNSUPPORTED QUERY scores 1.0") and penalises it with 0.00. This is a judge calibration issue, not a pipeline regression — the production behaviour is correct.

Full per-question scores, the v1 vs v2 delta analysis, and the Q13 judge false-negative are documented in [eval/eval_results.md](eval/eval_results.md).

## Limitations

1. **No result-level validation.** The validator catches structural issues such as unsafe keywords, wrong table, multiple statements but cannot catch SQL that is syntactically valid yet semantically wrong. An AND/OR precedence bug observed on Q3 returned an inflated count of 4,211 rows with no error. This is the most dangerous failure mode because it is invisible to the user.
2. **Non-deterministic SQL generation.** Running the same question twice can produce different SQL. Q3 produced a precedence error on one run and a correct query on the next. A single eval run does not establish correctness.
3. **Complex date range AND/OR logic errors.** Queries that span multiple years and months ("between March 2021 and March 2022") are prone to operator-precedence mistakes when the model omits parentheses. The v2 prompt rule that fixed this caused regressions on simpler month-only queries, so the rule needs to be scoped more precisely.

## Setup

1. Clone the repository.
   ```bash
   git clone https://github.com/<your-username>/ottawa-road-safety-analytics.git
   cd ottawa-road-safety-analytics
   ```
2. Create and activate a virtual environment.
   ```bash
   python -m venv venv
   venv\Scripts\activate         # Windows
   # source venv/bin/activate    # macOS/Linux
   ```
3. Install dependencies.
   ```bash
   pip install -r requirements.txt
   ```
4. Configure environment variables. Copy `.env.example` to `.env` and set `GROQ_API_KEY`.
5. Download the Ottawa Traffic Collision CSV from the City of Ottawa Open Data Portal and place it in `data/raw/Traffic_Collision_Data.csv`. See [docs/data_sources.md](docs/data_sources.md) for the exact link.
6. Run ingestion and preprocessing.
   ```bash
   python -m src.ingest
   python -m src.preprocess
   ```
7. Launch the Streamlit app.
   ```bash
   streamlit run dashboard/app.py
   ```

## Future Work

- **Result-level validation** to catch silent semantic failures (e.g. row counts outside plausible bounds, NULL-dominated GROUP BY results).
- **Map tooltips** are implemented — hover over a marker to see location, severity, date, impact type, and lighting condition.
- **UI improvements** — query history and keyboard shortcuts for example questions.
- **RAG retrieval quality** — fine-tune or replace `all-MiniLM-L6-v2` to close the semantic gap between natural-language time references ("night") and schema vocabulary ("Dark").
- **Multi-table support** — generalise the prompt and validator to join the collision table with future tables (e.g. road segments, traffic counts, weather).
